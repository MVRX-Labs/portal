## Problem Statement

MVRX Labs is a full-stack GTM agency. Clients pay ~$2/lead for curated lead lists targeting their ICPs. Today, list generation is a manual process: a human prompts Claude with web search and Apify MCP access, processes leads one-by-one, and copy-pastes results into spreadsheets. This doesn't scale — it requires constant human oversight, enrichment is slow (one row at a time), and quality is inconsistent (incomplete contact info, imperfect ICP fit, no systematic scoring).

Two entry points exist:

1. **From scratch** — client describes their ICP, MVRX builds the list from external data sources
2. **Enrich existing** — client provides a raw list (exported LinkedIn connections, email contacts), MVRX enriches and qualifies it

The goal is an automated pipeline where a team member defines the ICP, presses a button, and gets a scored, enriched lead list — with minimal human in the loop.

## Demand Evidence

- Already delivering lead lists to paying clients at ~$2/lead
- Multiple clients requesting lead lists as part of broader GTM engagements
- 100-1,000 leads per list is typical
- Clients need outbound-ready data: email, LinkedIn URL, and/or phone number

## Status Quo

Current process:

1. Human defines ICP parameters from client brief
2. Human prompts Claude (with web search + Apify MCP) to find leads
3. Claude searches, but results need manual verification
4. Enrichment is done row-by-row via Claude — extremely slow and token-heavy
5. No systematic quality scoring — human eyeballs ICP fit
6. Results copy-pasted into spreadsheet and delivered

Pain points (priority order):

1. **Manual process** — requires human oversight throughout
2. **Reliability/repeatability** — each list is artisanal, not a repeatable process
3. **Cost** — high token usage from row-by-row LLM processing
4. **Quality gaps** — incomplete contact info, imperfect ICP matching, no scoring

## Constraints

- Must work within existing MVRX Portal (Next.js + Drizzle + Trigger.dev)
- Must handle both entry points (from scratch + enrich existing)
- Output must be CSV/spreadsheet deliverable clients can load into outbound tools
- Must maintain or improve current quality at $2/lead price point
- LLM is orchestrator and scorer, not source of truth for lead data
- Apollo API is the primary structured data source (already planned in codebase)
- Apify LinkedIn scraping is already integrated in the codebase

## Premises

1. The primary output is a deliverable CSV — not a live dashboard or CRM integration
2. LLM orchestrates and scores, but real data sources (Apollo, LinkedIn, web) are the source of truth for lead data
3. Structured ICP definition is the main input
4. LLM quality scoring is the value-add that justifies $2/lead

## Approaches Considered

### Approach A: Structured Apollo Pipeline (Minimal Viable)

Predefined stages: Apollo search -> dedupe -> Apollo enrich -> batch LLM score -> CSV export. All Trigger.dev tasks, no agent reasoning. Single-source dependency limits coverage.

- **Effort:** M | **Risk:** Low
- **Pros:** Predictable, debuggable, cheap to run
- **Cons:** Apollo's data coverage varies by market; can't handle unusual ICPs

### Approach B: Multi-Source Waterfall Pipeline (Ideal Architecture)

Same staged pipeline but sources from Apollo + Apify LinkedIn scrape + web search. Waterfall enrichment. Quality gates. Both entry points as pipeline configurations.

- **Effort:** L | **Risk:** Medium
- **Pros:** Higher coverage, 85-90% contact info rates, quality gates prevent bad data
- **Cons:** More API subscriptions, more moving parts

### Approach C: Hybrid Agent + Deterministic Pipeline

Claude Agent SDK for the sourcing phase (agent decides tools based on ICP). Deterministic pipeline for enrichment, scoring, export.

- **Effort:** L | **Risk:** Medium-High
- **Pros:** Handles niche ICPs, adapts per-client without code changes
- **Cons:** Unpredictable agent behavior, higher token cost, harder to debug

## Recommended Approach

**Approach B: Multi-Source Waterfall Pipeline** — chosen because it gives reliability and coverage that justifies $2/lead without agent unpredictability. Apollo alone won't cover every ICP, and Apify is already integrated. Quality gates separate a professional deliverable from a spreadsheet of maybes.

## Architecture

### Pipeline Overview

```
[Job Created]
    |
    v
[Source Leads] -----> Apollo People Search API
    |                  Apify LinkedIn Scraper
    v
[Ingest / Dedupe] --> Merge sources, dedupe by LinkedIn URL + name
    |
    v
[Enrich] -----------> Apollo People Match (LinkedIn URL -> email/phone)
    |                  Email pattern guessing + MX validation (fallback)
    v
[Quality Gate] ------> Email deliverability check
    |                   LinkedIn URL validation
    |                   Minimum data completeness
    v
[ICP Score] ---------> Batch LLM scoring (20-50 leads per prompt)
    |                   Structured output: tier, confidence, rationale
    v
[Export] ------------> CSV generation
    |                  Store in DB
    |                  Slack notification with download link
```

### Entry Points

**From Scratch:**

1. Team member creates a job via Portal UI or API
2. Inputs: ICP definition (titles, industries, company sizes, geography, signals), target lead count, required contact modes (email/LinkedIn/phone)
3. Pipeline runs Source -> Dedupe -> Enrich -> Score -> Export

**Enrich Existing List:**

1. Team member uploads CSV with raw leads (name, company, LinkedIn URL, etc.)
2. Pipeline parses CSV, normalizes fields, deduplicates
3. Pipeline runs Enrich -> Score -> Export (skips sourcing)

**Note on `leads.linkedinUrl`:** The existing `leads` table has `linkedinUrl` as NOT NULL with a unique constraint on `(accountId, linkedinUrl)`. For CSV uploads where leads may not have LinkedIn URLs, the ingest task must attempt to find the LinkedIn URL (via Apollo match by name+company+email) before inserting. Leads that cannot be matched to a LinkedIn URL are stored with a unique placeholder (`placeholder_<cuid>`) to satisfy both constraints, and flagged for manual review.

**New table: `lead_list_job_leads`** (junction)

| Column             | Type                         | Description                      |
| ------------------ | ---------------------------- | -------------------------------- |
| id                 | text (cuid2, prefix `lljl_`) |                                  |
| jobId              | text (FK)                    | Lead list job                    |
| leadId             | text (FK)                    | Lead record                      |
| source             | text                         | Where this lead was sourced from |
| enrichmentStatus   | enum                         | `pending`, `enriched`, `failed`  |
| qualityGateStatus  | enum                         | `pending`, `passed`, `failed`    |
| qualityGateReasons | text[]                       | Why it failed, if applicable     |
| icpTier            | integer (nullable)           | 1/2/3                            |
| icpConfidence      | integer (nullable)           | 0-100                            |
| icpRationale       | text (nullable)              | LLM reasoning                    |
| includedInExport   | boolean                      | Whether it made the final cut    |

**Scoring lives on the junction table, not the lead.** The same lead can appear in multiple jobs with different ICP definitions, so scoring is per-job. The existing `leads.tier` / `leads.conversionPct` / `leads.rationale` fields (from inbound lead discovery) remain separate — they reflect the account-owner's ICP, not a lead list job's ICP. No migration needed; the two scoring contexts coexist.

### Failure Modes

Each pipeline stage has independent retry behavior:

- **Sourcing (Apollo/LinkedIn):** If one source fails but another succeeds, the pipeline continues with partial data. Job status becomes `completed_with_warnings` if any source failed. Only if ALL sources fail does the job fail at the sourcing stage.
- **Enrichment:** Retries per-batch (3 attempts with exponential backoff). Leads that fail enrichment after retries are marked `enrichmentStatus: 'failed'` but don't block the pipeline — they proceed to scoring with whatever data they have.
- **Scoring:** Retries per-batch. If a scoring batch fails, those leads get `icpTier: null` and are excluded from the export.
- **Quality gate:** No retries (deterministic checks). Leads that fail are marked with reasons and excluded from export.
- **Export:** If export fails, the job is marked `failed` and a Slack alert is sent. All prior work is preserved in the DB for re-export.

Minimum viable threshold: a job succeeds if it delivers at least 10% of `targetCount` leads. Below that, it's marked `failed` with a "insufficient leads" reason.

### Trigger.dev Task Structure

All tasks in `src/trigger/lead-list/`:

**1. `lead-list-orchestrator`** — Parent task

- Creates the job record
- Dispatches sourcing/ingestion based on entry point
- Chains: source -> dedupe -> enrich -> score -> quality-gate -> export
- Updates job status at each stage
- Sends Slack notification on completion or failure
- Tracks cost breakdown across stages

**2. `lead-list-source-apollo`** — Apollo People Search

- Takes ICP definition, converts to Apollo search filters
- Paginates through results (Apollo max 100 per page)
- Upserts leads into `leads` table + `lead_list_job_leads` junction
- Returns count of leads found

**3. `lead-list-source-linkedin`** — Apify LinkedIn Scraper

- Constructs LinkedIn Sales Navigator search URL from ICP parameters
- Triggers Apify actor (reuse existing `apifyLinkedinScraper` pattern)
- Parses results, upserts leads
- Queue-limited to avoid Apify rate limits

**4. `lead-list-ingest-csv`** — Client CSV Parser

- Accepts uploaded CSV
- Normalizes fields (name splitting, URL cleanup, company extraction)
- Deduplicates against existing leads by LinkedIn URL, then by name+company
- Creates lead records + junction entries

**5. `lead-list-enrich`** — Waterfall Enrichment (batch)

- Processes leads in batches of 50
- Step 1: Apollo People Match API (LinkedIn URL -> email, phone, region)
- Step 2: For leads missing email after Apollo — attempt lookup via company domain + name pattern (firstname@domain.com, f.lastname@domain.com). V1 uses common patterns + MX validation; v2 can add a dedicated enrichment API (Hunter.io, Snov.io)
- Updates lead records with enriched data
- Concurrency-limited queue to respect API rate limits

**6. `lead-list-score`** — Batch LLM Scoring

- Processes leads in batches of 20-30 per prompt
- Sends ICP definition + lead profiles to Claude
- Structured output: `{ tier: 1|2|3, confidence: 0-100, rationale: string }`
- Uses Haiku for cost efficiency (scoring is pattern matching, not deep reasoning)
- Updates junction table with scores

**7. `lead-list-quality-gate`** — Validation

- Email check v1: syntax validation + MX record lookup (confirms domain accepts email, does NOT confirm specific address). This is low-cost but low-confidence. Fast-follow: integrate a real verification service (ZeroBounce, NeverBounce) for SMTP-level validation if clients report bounce rates above 5%.
- LinkedIn URL validation (format check; optional Apify profile validation for high-value lists)
- Minimum completeness: must have at least one required contact mode
- Marks leads as passed/failed with reasons

**8. `lead-list-export`** — CSV Generation

- Filters to leads that passed quality gate
- Optionally filters by ICP tier (e.g., tier 1+2 only)
- Generates CSV with columns: name, title, company, LinkedIn URL, email, phone, region, ICP tier, ICP confidence, ICP rationale
- Stores CSV (reuse existing `leadCsvs` pattern)
- Updates job record with final counts
- Sends Slack notification with download link and summary stats

### ICP -> Apollo Filter Mapping

The ICP definition fields map to Apollo search parameters:

| ICP Field                                        | Apollo Parameter                         |
| ------------------------------------------------ | ---------------------------------------- |
| targetTitles                                     | `person_titles[]`                        |
| targetIndustries                                 | `organization_industry_tag_ids[]`        |
| targetCompanySizes                               | `organization_num_employees_ranges[]`    |
| targetSignals (person-level, e.g. "VP+")         | `person_seniority[]`                     |
| targetSignals (company-level, e.g. "Series A-C") | `organization_latest_funding_stage_cd[]` |
| targetSignals (keywords, e.g. "AI", "DevOps")    | `q_keywords`                             |

**Signal classification:** The existing `targetSignals` is a flat `string[]`. Before constructing Apollo queries, the orchestrator passes the array through a lightweight LLM call (Haiku) to classify each signal as `person`, `company`, or `keyword`. This avoids requiring schema changes or rigid naming conventions.
| targetGeographies (from job config) | `person_locations[]` |

### Cost Model

Per 1,000 leads (estimated):

| Stage                        | Cost                             |
| ---------------------------- | -------------------------------- |
| Apollo search                | ~$0.03/lead ($30 for 1K credits) |
| Apollo enrichment            | ~$0.03/lead                      |
| Apify LinkedIn scrape        | ~$0.01/lead                      |
| LLM scoring (Haiku, batched) | ~$0.005/lead                     |
| Email verification           | ~$0.005/lead                     |
| **Total COGS**               | **~$0.08/lead**                  |

**Note:** These are per-lead-sourced costs. Assuming a 75% quality gate pass-through rate (25% filtered for incomplete data or poor ICP fit), effective COGS per delivered lead is ~$0.11. At $2/lead selling price = **~94% gross margin** on delivered leads. Even with edge cases doubling costs, margins stay above 88%.

### API Routes

**`POST /api/accounts/[id]/lead-list-jobs`** — Create a new lead list job

- Body: `{ icpDefinitionId?, entryPoint, targetCount, contactModes[], sources[], inputCsv? }`
- Triggers the orchestrator task
- Returns job ID + Trigger.dev run ID

**`GET /api/accounts/[id]/lead-list-jobs`** — List jobs for account

- Paginated, filterable by status

**`GET /api/accounts/[id]/lead-list-jobs/[jobId]`** — Job detail with stats

- Includes real-time progress via Trigger.dev metadata

**`GET /api/accounts/[id]/lead-list-jobs/[jobId]/download`** — Download output CSV

### Portal UI

Minimal UI — this is an internal tool for MVRX team members:

1. **Job creation page** — Select account, choose/create ICP definition, set target count, pick sources, optionally upload CSV
2. **Job list page** — Table of jobs with status, progress, lead counts
3. **Job detail page** — Real-time progress (sourced/enriched/scored/exported counts), cost breakdown, download link when complete

## Success Criteria

1. A team member can create a lead list job and get a scored, enriched CSV without touching Claude manually
2. 80%+ of leads in the output have at least one valid contact method (email or LinkedIn URL)
3. ICP scoring rationale is specific enough that a client can understand why each lead was included
4. End-to-end time for a 500-lead list is under 30 minutes
5. COGS per lead is under $0.20 (10x margin at $2/lead)
6. Both entry points (from scratch + enrich existing) work through the same pipeline

## Dependencies

- **Apollo API subscription** — needed for people search and enrichment
- **Email verification service** — for quality gate
- **Existing infrastructure** — leads table, ICP definitions, Apify integration, CSV export logic all exist and can be reused
- **No new external services beyond Apollo** — Apify and Claude are already integrated

## The Assignment

Before building anything: **run the pipeline manually for one real client using the exact stages described above — but do each step by hand.** Use Apollo's web UI to search, manually run enrichment queries, score 20 leads in a single Claude prompt with structured output, and validate emails. Time each stage. This will tell you:

1. Whether Apollo's filters actually map to your clients' ICPs
2. What the real enrichment hit rate looks like
3. Whether batch LLM scoring produces useful tier/rationale output
4. Where the actual bottlenecks are vs. where you assume they are

Do this for ONE client list before writing any code. The pipeline design above assumes Apollo coverage is good and batch scoring works — validate those assumptions with real data first.
